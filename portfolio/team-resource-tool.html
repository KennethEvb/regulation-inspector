The problem

The team planner is always out of sync with the project board.

Most resource planners live in static spreadsheets owned by individual team leads. These trackers fall out of sync with the actual team rosters in HR systems, and they never match the task-level reality of JIRA boards. This leads to double-booking, missed deadlines, and burnt-out teams. This concept case study explores a unified Power Platform architecture that links IAM rosters, FTE allocations, and initiative assignments in a single Dataverse model, while presenting a visual capacity planner that changes based on the maturity of the organisation's tooling.

The approach

A scalable blueprint for team capacity

01

IAM synchronization

Integrate with Azure Active Directory (Microsoft Entra ID) to pull the true team roster — automatic starters, leavers, and department changes.

02

Unified allocation model

Team members are allocated to initiatives as fractional FTEs (e.g. 0.5 FTE on Initiative A) rather than simple names on cards, with enforced validation to prevent overallocation.

03

FTE forecasting

Power Automate compiles the allocations into a monthly timeline, generating a visual heat map that identifies resource shortages before they impact the schedule.

04

Dual-mode scaling

The system is built to scale: start with manual entry (MVP) to establish the habit and data model, then plug in the JIRA Cloud connector (Enterprise) to pull actual task backlogs automatically.

Data model

A robust relational schema in Dataverse.

Rosters, initiatives, allocations, and demands are held in six related tables with enforced keys and validation rules. Double-booking gets caught at the database level, before a user can even save the record.

Reflection

Decisions that held up

Design for modular growth

The data layer was separated from the interface from day one, so we can replace manual entry with automated JIRA APIs without rebuilding the planner UI or the Power BI reports.

Enforce rules at the database layer

Trying to prevent overallocation inside individual form screens is brittle. Creating validation rules directly in Dataverse ensures data integrity, no matter which app writes the record.

Capacity planning is a timing problem

Rosters change, holidays happen, and projects slip. A capacity model that assumes a fixed team over a six-month window is immediately wrong. The design incorporates holiday logs directly into available capacity calculations.

Actionable alerts are better than complex reports

Team leads don't want to dig through dense reports to find over-allocated people. Visual highlights (like David Kim's red alert) tell them exactly where the bottleneck is in three seconds.
